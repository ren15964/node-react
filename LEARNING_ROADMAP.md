# 全栈学习路线图

## 项目定位

这个项目不只是一个练手项目，它要逐步成长为一个可以用于求职展示的全栈项目，用来同时支撑这几件事：

- 逐步学习 React 和 Node.js
- 理解企业开发中的常见工程习惯
- 为面试做准备
- 在简历中体现你的工程思维

当前技术栈：

- 前端：React、Vite、React Router、Ant Design、Axios
- 后端：Node.js、Express、MySQL、JWT、Joi、Multer

当前已经有的能力：

- 用户注册 / 登录
- 路由守卫
- 文章列表、新增、编辑、删除、恢复
- 分页和关键词搜索
- 图片上传

当前主要短板：

- 缺少清晰的项目启动脚本和启动流程
- 工程结构还不够稳定，文档也不够完整
- 错误处理还不够统一
- 没有测试覆盖
- 权限设计和安全设计还比较弱
- 业务深度还不够，离“可展示的项目”还有距离

## 我们之后如何协作

后面每做一个功能或重构，都尽量按这个学习闭环推进：

1. 先明确目标和业务意义。
2. 用通俗中文解释相关概念。
3. 小步实现，不一次改太大。
4. 改完后解释代码在干什么。
5. 做验证，确认功能真的通了。
6. 说明当前实现有哪些简化，企业里通常还会怎么做。

这样做的目的不是只把功能“堆出来”，而是让你真正理解代码为什么这样写。

## 第 0 阶段：先把当前项目打稳

目标：

- 让当前项目具备继续迭代的基础，不至于越写越乱

要做的事：

- 给前后端补齐规范的 `dev` / `start` 启动脚本
- 增加 `.env.example`
- 清理误加入的文件和敏感信息暴露问题
- 修复代码和界面里的乱码文案
- 统一基础 README 和启动说明
- 检查接口返回结构是否一致

为什么重要：

- 企业非常看重一个项目是否能被稳定启动、理解和交接
- 一个连启动都不顺手的项目，很难成为有说服力的简历项目

这一阶段你会接触的面试点：

- 环境变量
- 项目启动流程
- 配置与代码分离
- 基础工程规范

## 第 1 阶段：夯实后端基础

目标：

- 让后端从“能跑的 CRUD 服务”提升到“更像真实服务”

要做的事：

- 明确 route、controller、validator、service 的职责边界
- 优化错误处理和错误响应规范
- 更严格地校验 params、query、body
- 让事务真正用于多步数据库写入的业务场景
- 统一响应工具和错误类
- 增加基础请求日志能力

你会学到：

- Express 请求生命周期
- 中间件执行顺序
- 为什么校验应该发生在业务逻辑之前
- 什么情况下需要事务
- 分层为什么能提升可维护性

企业开发视角：

- 真实服务要有稳定、可预测的错误响应格式
- 参数校验不是“可选项”，它是 API 契约的一部分

面试重点：

- 中间件机制
- 一次请求在后端里的流转过程
- controller 和 service 的职责区别
- 事务使用场景

## 第 2 阶段：升级鉴权和权限设计

目标：

- 从“能登录”升级到“有基本可用的鉴权方案”

要做的事：

- 改善 token 的处理方式和登录态持久化策略
- 提供更完整的当前用户信息
- 在合适的地方引入角色或资源归属概念
- 明确谁可以新增、编辑、删除、恢复文章
- 把登录用户信息写入文章业务中
- 先建立 refresh token 的认知，即使当前阶段未完整实现

你会学到：

- 认证和授权的区别
- JWT payload 基础
- 为什么前端路由守卫远远不够
- 为什么权限校验必须在后端做

企业开发视角：

- 企业系统往往更重视“谁有权限做什么”，而不只是“能不能登录”

面试重点：

- 鉴权流程的完整链路
- token 存储方式的取舍
- 为什么后端必须做权限校验

## 第 3 阶段：把文章模块做成像样的业务模块

目标：

- 让文章模块不只是 CRUD，而是一个完整业务模块

要做的事：

- 增加文章详情页，而不是点击标题直接进入编辑
- 支持草稿 / 已发布等状态
- 增加作者信息和归属展示
- 引入分类或标签设计
- 支持更丰富的搜索和排序
- 改进软删除后的管理视图
- 后续可以逐步引入 Markdown 或富文本编辑器

你会学到：

- 业务建模
- 列表页、详情页、编辑页的职责区别
- 表结构如何随着业务演进
- 表单、表格、接口之间如何配合

企业开发视角：

- 一个能写进简历的项目，往往不是看你按钮有多少，而是看模块生命周期是否完整

面试重点：

- 数据建模
- 软删除设计
- 草稿 / 发布流程
- 后台管理模块设计

## 第 4 阶段：升级前端工程结构

目标：

- 让前端代码更容易扩展，也更适合面试时讲清楚

要做的事：

- 按模块拆分 API 方法，不要在页面里到处直接写 Axios
- 区分页组件和可复用业务组件
- 优化表单状态和错误反馈
- 增加 loading、empty、error 等状态处理
- 优化路由结构和导航逻辑
- 抽出时间格式化等通用工具

你会学到：

- 为什么组件边界重要
- 为什么接口封装能减少重复代码
- 一个页面如何组织数据、UI 和导航逻辑

企业开发视角：

- 企业前端大量工作并不是“画界面”，而是控制复杂度、提高可维护性

面试重点：

- 组件设计
- 请求封装
- 状态管理选型
- 用户体验兜底状态

## 第 5 阶段：加入测试和质量保障

目标：

- 证明这个项目是“工程化项目”，不是“拼装起来能跑”

要做的事：

- 为鉴权和文章流程增加后端接口测试
- 为关键页面增加前端基础测试
- 完善 lint、format 等质量检查脚本
- 定义每次改动至少要做哪些验证

你会学到：

- 为什么测试能保护重构
- 单元测试和集成测试的差异
- 如何更有信心地验证 API 行为

企业开发视角：

- 团队依赖质量门禁，因为手工测试无法长期扩展

面试重点：

- 测试分层思想
- 集成测试
- 如何防止回归问题

## 第 6 阶段：补上安全与运维基础意识

目标：

- 学到企业里真正关心的非功能性要求

要做的事：

- 增强上传安全校验和文件命名策略
- 收紧 CORS 和请求体限制
- 讨论密码策略和暴力破解防护思路
- 规范密钥管理方式
- 视情况增加审计相关字段
- 补齐部署变量和生产注意事项文档

你会学到：

- 安全不是最后才补的，而是从设计阶段就要考虑
- 上传和登录往往是常见攻击面
- “我本地能跑”不等于“能上线”

面试重点：

- CORS
- 密钥管理
- 上传安全
- 请求校验与防滥用

## 第 7 阶段：做简历和面试包装

目标：

- 让这个项目以“应聘者项目”的标准呈现出来

要做的事：

- 写清晰的项目介绍
- 说明架构、模块和设计取舍
- 准备截图和演示流程
- 整理成可以写进简历的技术亮点
- 准备项目中你解决过的问题和思考过程

你最终应该能讲清楚：

- 项目整体架构
- 登录鉴权链路
- 文章模块设计
- 为什么用软删除
- 参数校验和错误处理怎么组织
- 前后端如何协作
- 如果进入生产环境，你下一步会继续补什么

## 推荐执行顺序

这个仓库后续最合理的推进顺序是：

1. 第 0 阶段：先把当前项目打稳。
2. 第 1 阶段：夯实后端基础。
3. 第 2 阶段：升级鉴权和权限设计。
4. 第 3 阶段：把文章模块做深。
5. 第 4 阶段：升级前端工程结构。
6. 第 5 阶段：加入测试和质量保障。
7. 第 6 阶段：补上安全和运维基础意识。
8. 第 7 阶段：做简历和面试包装。

## 当前建议的第一轮 Sprint

下一轮建议优先做基础建设，顺序如下：

1. 给前后端加启动脚本，整理项目启动流程。
2. 增加 `.env.example`，规范敏感信息处理。
3. 修复乱码文案，统一接口返回文案。
4. 优化文章和鉴权模块结构。
5. 增加当前用户接口，完善登录态处理。

为什么现在先做这些：

- 改动规模合适，适合学习阶段吸收
- 能快速提升项目可信度
- 能给后面继续加功能打一个更稳的基础

## 我们的协作约定

后面继续做这个项目时，我应该做到：

- 用适合初学者理解的中文解释每一次改动
- 告诉你这段代码解决了什么问题
- 把它和企业里的真实做法联系起来
- 明确指出当前实现是简化版还是更接近生产
- 帮你建立能面试表达的理解，而不只是把页面做出来

你需要尽量做到：

- 遇到概念不懂时及时追问
- 尝试用自己的话复述一次请求流或数据流
- 一次先把一个模块理解透，再继续叠功能
- 让项目以小步、可解释的方式持续推进
